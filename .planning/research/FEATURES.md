# Feature Landscape

**Domain:** Berlin Gymnasium comparison and directory website
**Researched:** 2026-03-26
**Overall confidence:** MEDIUM — core features verified against GreatSchools, Niche.com, gymnasium-berlin.net, Tagesspiegel interactive map, JedeSchule.de; Berlin-specific admission flow confirmed via official berlin.de sources.

---

## Table Stakes

Features users expect. Missing = product feels incomplete or useless compared to existing alternatives.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| School listing with name, address, district | Core directory — any school site has this | Low | From Berlin Open Data WFS / schulverzeichnis |
| Filter by district (Bezirk) | Berlin parents think geographically; 12 districts | Low | Enum filter; data always present |
| Filter by school profile/specialization | MINT, music, sports, bilingual, altsprachlich — primary selection criterion | Low | Enum multi-select; 6-7 profile categories |
| Filter: accepts after 4th grade (grundständig) | The defining differentiator for this project; ~30 of 90+ schools | Low | Boolean flag; must be visually prominent |
| School detail page | Parents need to deep-dive one school | Medium | Per-school page or modal with all data fields |
| Contact info on detail page | Parents call/email to arrange visits | Low | Phone, email, website link |
| Interactive map of all schools | Spatial orientation is how parents think about commute | Medium | Standard Leaflet/MapLibre; coordinates required |
| Click map pin → school detail | Map is useless without this connection | Low | Linking/modal trigger |
| Ratings display with source attribution | Trust signal; parents expect quality signals like Google rating | Medium | Must show source (Google Maps, etc.) not just number |
| Foreign languages offered | Bilingual and language-focused families filter on this | Low | Enum multi-select; data available from schulverzeichnis |
| Ganztag / half-day flag | Working parents cannot choose half-day schools | Low | Boolean; critical for dual-income families |
| School website link | Parents will always go to the school's own site | Low | Direct link out |
| Mobile-responsive layout | >60% of initial browsing is mobile | Medium | Static SPA must work on small screens |
| Fast initial load | Parents bounce on slow sites; WASM bundle size needs attention | High | WASM cold-start latency is a known Leptos issue; critical to address |

## Differentiators

Features that set this product apart from gymnasium-berlin.net, schulen.de, and Tagesspiegel. Not expected, but highly valued.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Travel time from entered address (walk, transit, car) | The single most actionable decision factor for most parents; no existing Berlin school site offers this | High | Client-side OSRM call; user enters address, results geocoded, routing called per school; cache results to avoid re-calling for all 90 schools |
| Sort/filter by travel time | Meaningless to show times unless parents can rank by them | Medium | Depends on travel time feature; UI sort + filter threshold |
| Side-by-side school comparison (2-3 schools) | Parents always shortlist 2-3; Niche.com comparison is highly used | Medium | URL-encoded school IDs; comparison table view |
| Admission requirements per school | Notendurchschnitt threshold, oversubscription ratio, selection criteria — unavailable in one place today | High | Data quality/availability is the hard part; display is straightforward |
| Open day (Tag der offenen Tür) dates | Time-sensitive, high-value; parents miss open days because no central list exists | Medium | Staleness is a risk — must show data timestamp; best-effort from school websites |
| Student-teacher ratio | Proxy for class size / teaching quality; parents trust it | Low | Display only; ratio calculated from student/teacher counts |
| Aggregated multi-source rating | More trustworthy than single Google rating; unique if built well | Medium | Requires scraping multiple sources; display with per-source breakdown |
| IB / bilingual track / honors program flag | Growing demand from expat and ambitious-parent segment | Low | Boolean flags + description field |
| Permalink / shareable URL per school | Parents share links in WhatsApp groups; gym-berlin.net lacks clean URLs | Low | URL routing in Leptos SPA; school slug in URL |
| Data freshness indicator | Parents need to know if data is current; trust signal | Low | Last-updated timestamp from pipeline run; shown on page |
| Changelog between pipeline runs | Transparency about what changed; useful for returning visitors | Medium | Generated by scraping pipeline; surfaced in UI as "what's new" |

## Anti-Features

Features to explicitly NOT build. Each has a clear reason and a better alternative.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| User accounts / login | Adds backend complexity, GDPR surface area, zero benefit for static read-only use case | Keep fully stateless; let bookmarks/shared URLs do the job |
| User reviews / comments | Moderation burden, legal exposure, high spam risk; existing review platforms (Google Maps) already capture this | Aggregate and link to existing reviews with source attribution |
| Real-time data | No backend, no SSE, no WebSockets possible on GitHub Pages static deployment | Clear "data as of [date]" label; re-run pipeline when schools update info |
| Enrollment / application form | Schools handle their own enrollment; redirecting parents there is cleaner | Deep-link to school's own Anmeldung page |
| School rankings / league tables | Abiturnoten ranking (Tagesspiegel already does this) invites controversy and is not the product's goal; parents choosing school for their 4th-grader care about fit, not rank | Show raw data (Abitur average if available); let parents decide weight |
| Recommendation engine / AI match | Scope creep; adds complexity without reliable data foundation | Clean filters + comparison table serves the same need deterministically |
| Email notifications / alerts | Requires backend and user registration | "Check back" message + RSS/Atom feed if needed later |
| Catchment area polygons | Berlin Gymnasien are not catchment-based (parents choose freely); would mislead users | Travel time radius is the correct mental model |
| School forum / Q&A | Content moderation, cold-start problem; better platforms exist | Link to relevant Facebook groups, Berlin expat forums |
| Tuition display | All Berlin Gymnasien are public / free; showing "0" is confusing | Omit the field entirely |

## Feature Dependencies

```
Travel time display
  ← Address geocoding (Nominatim/Photon)
  ← OSRM routing API (client-side HTTP calls)
  ← School coordinates (lat/lng in YAML data)

Sort/filter by travel time
  ← Travel time display (must compute first)

Side-by-side comparison
  ← School detail page (reuses same data model)
  ← Shareable URL / permalink (encodes selected schools)

Aggregated multi-source rating
  ← Scraping pipeline (Google Maps, schulen.de, etc.)
  ← Per-source attribution display

Open day dates
  ← Scraping pipeline (school websites)
  ← Data freshness indicator (staleness risk)

Admission requirements detail
  ← Scraping pipeline (school websites + official berlin.de)
  ← Grundständig flag (already required as table stakes)

Interactive map
  ← School coordinates (lat/lng)
  ← School listing (all schools must be loaded)

Filter by travel time
  ← Travel time computed for all visible schools
  ← Performance: must batch OSRM calls, not serial
```

## MVP Recommendation

Ship in this order to validate core value before building differentiators:

**MVP (Phase 1-2):**
1. School listing with name, district, profile, grundständig flag
2. Filter by district, profile, grundständig
3. School detail page (all scraped fields)
4. Interactive map with school pins
5. Mobile-responsive layout

**Phase 3 (core differentiator):**
6. Travel time from user-entered address (walk + transit + car)
7. Sort by travel time
8. Side-by-side comparison (2-3 schools)

**Phase 4 (data depth):**
9. Open day dates (with freshness date)
10. Admission requirements per school (Notendurchschnitt, criteria)
11. Aggregated multi-source ratings

**Defer indefinitely:**
- Changelog UI (pipeline produces it; surfacing in UI is polish)
- IB / honors track flags (data availability uncertain)
- Data freshness per-field (overhead without proven need)

## Berlin-Specific Context Notes

These factors are unique to this domain and inform feature priority:

1. **4th vs 6th grade entry is the central decision.** Only ~30 of 90+ schools are grundständig. This flag must be the most prominent filter — it is the primary reason parents would choose this site over gymnasium-berlin.net.

2. **Admission is competitive and process-driven.** Parents need: Notendurchschnitt threshold, oversubscription ratio, whether a Probeunterricht (trial lesson) is required, and whether there is an entrance test. This is not available in any single clean source today.

3. **Open days are time-critical.** Berlin's Anmeldezeitraum (registration window) is a 3-4 day window in February. Parents must attend a Tag der offenen Tür before then to make an informed choice. Missing an open day closes options. A calendar of upcoming open days with school name + date is extremely high value.

4. **No catchment restrictions.** Berlin parents can apply to any Gymnasium citywide. Travel time (not catchment area) is the right spatial feature.

5. **Language tracks drive choice.** Altsprachlich (Latin/Ancient Greek), bilingual (English, French, etc.), and MINT profiles are the main filter axes after location. The foreign language sequence (from which grade) matters.

6. **Google Maps ratings exist but are thin.** Most Berlin Gymnasien have 10-50 Google reviews. Aggregating across sources adds marginal signal. Show them with attribution but don't over-invest in aggregation logic early.

7. **Tagesspiegel already publishes Abitur rankings.** Don't replicate their rankings feature. Link to their interactive map instead. Focus on admission-stage information, not outcome-stage information.
